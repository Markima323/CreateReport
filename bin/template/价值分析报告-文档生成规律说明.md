# 价值分析报告文档生成规律说明

## 1. 分析范围

对比了 1 份原始模板、5 份已完成报告、身份证/不动产登记簿图片和历史数据。成品之间的差异可归为五类：项目级常量、图片直接提取、补充 JSON、格式化派生值、条件性章节与计算判断。

## 2. 关键结论

- 这不是单纯的“黄色文字全部替换”任务。模板中存在完整章节删除、同级标题重编号、债务人数量分支、安置费用二选一、结论最低值名称判断和身份证校验。
- 文件名使用支行与户名；报告编号使用序号。
- 正文预计清收日期使用 JSON 的“预计回收”字段，不使用“清收完成时间”。
- 债权评估值取“基准日本息、权利价值、可回收金额”三者最低值，结论中的名称必须跟随实际最低项变化。
- 安置费用为 0/空时，公式和计算式都不能出现安置费用或 `-0.00`。
- 保证人为空时，应删除“债务责任关联方简介”整个章节，并对后续章节编号前移。
- 债务人只有一人时写“债务人”；多人时按“债务人一、债务人二……”编号。
- 地址末尾室号用于推导楼层，例如 406 室为 4 层，3-402 室为 4 层。
- 身份证提取后仍需执行居民身份证校验；17 位数字允许按校验位规则受控补齐。

## 3. 数据层级

### 项目级参数

项目年、报告年、报告批次、分析基准日、报告日期、报告日期大写、有效期截止日和折现率不应散落在 Prompt 中，应作为每次任务的显式参数传入。有效期截止日不能从现有样例可靠推导，应由调用方给定。

### 行级字段

每个人员图片文件夹生成一条 JSON 记录和一份报告。文件夹名提供序号、支行和姓名；登记簿图片提供坐落、面积、权证编号、权利人、登记类型和抵押债权金额；身份证图片提供债务人身份信息。图片中不存在的估值字段由明确的补充 JSON 提供。

### 派生字段

包括万元金额、百分比、民族显示、债务人称谓与连接字符串、楼层、安置费用段落、回收金额公式文字、最低值名称、文件名和报告编号。

## 4. 计算关系

- 基准日本息 = 基准日本金 + 基准日利息
- 评估价值 = ROUND(面积 × 评估单价, 0)
- 快速变现价值 = ROUND(评估价值 × 快速变现系数, 0)
- 折现系数 = ROUND((1 + 折现率)^(-折现期限), 4)
- 折现价值 = ROUND(快速变现价值 × 折现系数, 0)
- 可回收金额 = 折现价值 - 处置费用 - 安置费用
- 债权评估值 = MIN(基准日本息, 权利价值, 可回收金额)
- 偿债率 = 债权评估值 ÷ 基准日本息

输入 JSON 已有计算值作为报告值，同时由程序独立复核；不一致时输出警告，不让模型静默覆盖。

## 5. 推荐程序结构

1. 遍历 `InputPic` 的人员子文件夹，将文件夹内全部图片以一次多图请求发送给 Gemini 3.5 Flash，并以高分辨率模式读取。
2. 输出合并 JSON 和每个人的独立 JSON；与补充数据合并后检查 `missing_fields`。
3. 进行公式、身份证、必要字段和地址楼层校验。
4. 面板识别图片 JSON 中的唯一保证人后，用户自行查询并粘贴原始企业资料。Gemini 仅从粘贴文本中提取企业字段并写入规则 JSON。
5. 由确定性代码修改 DOCX 的 OOXML，而不是让模型重新排版文档。
6. 删除可选段落/表格行，更新域、目录和页码，并对生成文件执行 XML 扫描与数值复核。

## 6. 模板说明

提供的基底模板沿用原模板版式，黄色高亮为普通替换字段，蓝色高亮为条件性字段。生成完成后必须移除全部高亮和未使用段落。模板中加入了债务人 3、债务人 4 的可选段落。程序通过 Microsoft Word 后台更新目录和页码，并保留 `TOC`、`PAGEREF`、`PAGE` 等域代码；模板不设置“打开文档时自动更新域”，避免 Word 弹出外部字段更新提示。
